# День 39 — Сборка мусора и счётчик ссылок

## 1. Reference Counting

В CPython каждый объект хранит счётчик ссылок.

Если счётчик становится равным 0 — объект уничтожается немедленно.

Важно понимать:

- Переменная — это ссылка на объект.
- Присваивание не копирует объект, а увеличивает число ссылок.
- Контейнеры (list, dict и т.д.) тоже хранят ссылки.
- Передача объекта в функцию временно увеличивает счётчик.

Для отладки можно использовать:

```python
import sys
sys.getrefcount(obj)
```

Значение будет на 1 больше из‑за временной ссылки аргумента.

---

## 2. Время жизни объекта

Объект живёт, пока существует хотя бы одна ссылка на него.

Удаление имени (`del a`) удаляет только ссылку из текущего пространства имён, а не сам объект.

Объект уничтожается только тогда, когда:

- нет ссылок
- refcount == 0

---

## 3. Циклические ссылки

Если объекты ссылаются друг на друга, их счётчики ссылок никогда не станут 0.

Пример:

```python
class Node:
    def __init__(self):
        self.ref = None

 a = Node()
 b = Node()
 a.ref = b
 b.ref = a
```

Даже после `del a, del b` объекты продолжают держать друг друга.

---

## 4. Циклический сборщик мусора

Python имеет дополнительный GC, который:

- ищет недостижимые циклы
- удаляет их

Он работает поверх reference counting.

Принудительный запуск:

```python
import gc
gc.collect()
```

GC удаляет только объекты, которые недостижимы из корней (глобальные переменные, стек вызовов и т.п.).

---

## 5. Почему нельзя полагаться на **del**

Метод `__del__` вызывается при уничтожении объекта, но:

- при циклах он может не вызваться вовремя
- порядок удаления объектов не гарантирован
- поведение не является надёжным механизмом управления ресурсами

Для управления ресурсами используется `with` (контекстные менеджеры), а не сборка мусора.

---

## Ключевая ментальная модель

1. Python сначала использует reference counting.
2. Если возникает цикл — подключается циклический GC.
3. Объект удаляется, когда он недостижим и не имеет ссылок.

Понимание этой модели критично для:

- кешей
- долгоживущих сервисов
- работы с большими структурами данных
- правильного проектирования классов
