## Задача 1 — Подсчёт ссылок

Объясни, какие ссылки существуют на объект `lst` на каждом шаге.

```python
import sys

 def test():
     lst = [1, 2, 3]
     print(sys.getrefcount(lst))
     return lst

 a = test()
 b = [a]
 del a
```

Вопросы:

1. Почему значение внутри функции больше ожидаемого?
2. Будет ли объект уничтожен после `del a`?
3. Когда он реально будет удалён?

## Задача 2 — Циклическая ссылка

```python
import gc

class Node:
    def __init__(self):
        self.ref = None

 a = Node()
 b = Node()
 a.ref = b
 b.ref = a

 del a
 del b
```

Вопросы:

1. Почему объекты не удаляются сразу?
2. Что именно удерживает их в памяти?
3. Что произойдёт после `gc.collect()`?

---

## Задача 3 — Проверка понимания

Ответь письменно:

1. В чём разница между “refcount == 0” и “объект недостижим”?
2. Почему reference counting не может решить проблему циклов?
3. Почему управление ресурсами нельзя строить на `__del__`?

---

# Решения

## Решение 1

- Внутри функции есть локальная ссылка `lst`.
- `getrefcount` создаёт временную ссылку.
- После возврата остаётся ссылка `a`.
- После помещения в список появляется ссылка `b[0]`.
- После `del a` объект живёт, потому что остаётся ссылка из `b`.
- Он будет удалён, когда будет удалён `b` или элемент из него.

## Решение 2

- После `del a` и `del b` внешние ссылки исчезают.
- Но объекты держат друг друга через `.ref`.
- Их refcount не равен 0.
- Циклический GC найдёт и удалит их при `gc.collect()`.

## Решение 3

- refcount == 0 означает отсутствие ссылок вообще.
- Недостижимость означает отсутствие ссылок из корней, но внутри цикла ссылки могут существовать.
- Reference counting не видит структуру графа, он считает только число ссылок.
- `__del__` ненадёжен, потому что его вызов зависит от механики GC и порядка удаления объектов.
